### 2.2 Git/ブランチ運用ルール

チーム開発の混乱を避けるため、以下のルールを厳守してください。

1.  **メインブランチ保護:** `main` ブランチには**直接コミットを禁止**します。
2.  **ベースブランチ:** **常に `develop` ブランチ**から作業用ブランチを作成してください。
3.  **ブランチ命名規則:** 以下の形式を厳守してください。
    - `feature/[担当者名]/[タスク概要]`
    - **(例)** `feature/studentA/nav-bar-fix`、`feature/studentB/add-speaker-profiles`

#### プルリクエスト (PR) の運用

- **PR の提出:** 作業が完了し、`develop`へのマージを希望する場合にのみ提出してください。
- **Issue 連携:** PR を出す際は、必ず**関連する Issue 番号を本文に記載**し、マージ時に Issue が自動で閉じられるようにしてください。
  - **(例)** `close #123` `Fixes #123`
- **マージ権限:** PR のレビューとマージは、**[髙畑 匠]（リーダー）または [小林 煌希]（Git 運用担当）**のみが行います。

### 2.3 スタイルガイドライン（デザインルール）

一貫したデザインを保つため、以下のスタイル定義に従って実装してください。

| 要素                  | 指定                                                                                            |
| :-------------------- | :---------------------------------------------------------------------------------------------- |
| **メインカラー**      | `#007bff` (Bootstrap の Primary Blue を想定)                                                    |
| **フォント**          | 全体を通じて `sans-serif` を使用                                                                |
| **見出し 1 (`<h1>`)** | サイズ: `32px`、ウェイト: `bold`                                                                |
| **スペーシング**      | **要素間のマージン・パディング**は **`2%`の倍数**（`2%`, `4%`, `6%`, `8%`など）で統一すること。 |

## 4. 作業開始手順

1.  リポジトリをクローン: `git clone [リポジトリURL]`
2.  最新の `develop` ブランチに移動: `git checkout develop`
3.  Issue を確認し、担当タスクに合わせたフィーチャーブランチを作成する。

### 日常の開発ルール

- **作業開始時:** 毎日作業を始める前、または大きなタスクに着手する前に、必ず**`develop`ブランチに切り替え**て**`git pull origin develop`**を実行し、ローカルを最新の状態に保ってください。
- **コンフリクト回避:** 最新のコードで作業することで、他のメンバーの変更を見落とすことによる**マージ時のコンフリクトを最小限**に抑えられます。
